# 全中文数字符号与后缀句法自举计算体系构造研究

**摘要**  
本文提出并构造了一套完全独立于西方数字符号（0-9, A-F）及英文助记符的计算系统自举方案。方案以中文数字符号 `〇~〥` 为十六进制表示基础，采用后缀式（Forth 哲学）中文句法描述指令操作，从裸金属机器码层面逐步实现 Hex0 加载器、宏汇编器直至自持开发环境。文中详细定义了数字代码映射表、寄存器命名规格、指令操作符集合及汇编器两遍扫描算法，并通过多维表格和流程图示展示了从二进制电信号到自举编译器的完整链条。该体系为建立文化自主、可审计的计算机底层生态提供了理论参考与工程原型。

**关键字**：自举；中文数字符号；后缀句法；Hex0；可信任构造；汇编器设计

---

## 一、引言

现代计算机体系自诞生之初便与拉丁字母及阿拉伯数字深度绑定。从机器码的十六进制表示（0-9, A-F）到汇编助记符（MOV, ADD, JMP），西方符号系统构成了软件栈的底层符号基础。这一现状带来了两个层面的问题：其一，非拉丁文化背景的学习者需额外克服符号认知障碍；其二，在可信任构造（Trusting Trust）防御场景中，审查 ASCII 源码仍无法规避编译器二进制后门的潜在风险——因为审查者必须信任把 ASCII 源码转换为二进制的工具链本身。

本文提出一种彻底的替代方案：**从二进制电信号表示开始，全部采用中文符号重写计算体系的数字与指令层**。具体而言：

1. **数字层**：用 `〇` 和 `一` 表示二进制位，用 16 个中文/苏州码子符号 `〇~〥` 表示十六进制数字。
2. **指令层**：用中文功能描述字（如 `累加器`、`移`）替代英文寄存器名和助记符，并采用后缀句法以降低解析复杂度。
3. **工具链层**：手工构造一个仅接受 `〇~〥` 符号的 Hex0 加载器作为信任根，逐步衍生出宏汇编器，最终实现自举。

本文会系统阐述上述体系的设计规格、关键算法与自举流程。

## 二、中文数字符号体系设计

### 2.1 二进制位符号定义

计算机硬件以高低电平表示逻辑状态。本体系定义：

| 逻辑状态 | 物理含义 | 符号 | Unicode 码点 |
|:---|:---|:---|:---|
| 低电平 | 0 | `〇` | U+3007 |
| 高电平 | 1 | `一` | U+4E00 |

> **创新说明**：用独立象形符号 `〇` 与 `一` 取代阿拉伯数字 `0` 和 `1`，使二进制表示彻底汉化。

### 2.2 十六进制数字集定义

每 4 位二进制序列（`〇一` 串）缩略为一个十六进制符号。集合定义如下：

| 数值 | 二进制 | 符号 | 名称/来源 | Unicode | 数值 | 二进制 | 符号 | 名称/来源 | Unicode |
|:---|:---|:---|:---|:---|:---|:---|:---|:---|:---|
| 0 | `〇〇〇〇` | 〇 | 零 | U+3007 | 8 | `一〇〇〇` | 八 | 八 | U+516B |
| 1 | `〇〇〇一` | 一 | 一 | U+4E00 | 9 | `一〇〇一` | 九 | 九 | U+4E5D |
| 2 | `〇〇一〇` | 二 | 二 | U+4E8C | 10 | `一〇一〇` | 十 | 十（重载） | U+5341 |
| 3 | `〇〇一一` | 三 | 三 | U+4E09 | 11 | `一〇一一` | 〡 | 苏州码子 1 | U+3021 |
| 4 | `〇一〇〇` | 四 | 四 | U+56DB | 12 | `一一〇〇` | 〢 | 苏州码子 2 | U+3022 |
| 5 | `〇一〇一` | 五 | 五 | U+4E94 | 13 | `一一〇一` | 〣 | 苏州码子 3 | U+3023 |
| 6 | `〇一一〇` | 六 | 六 | U+516D | 14 | `一一一〇` | 〤 | 苏州码子 4 | U+3024 |
| 7 | `〇一一一` | 7 | 七 | U+4E03 | 15 | `一一一一` | 〥 | 苏州码子 5 | U+3025 |

> **创新说明**：  
> 1. 把汉字数字 `十` 重载为数值 10（A），苏州码子 `〡〢〣〤〥` 重载为 11~15（B~F），补足十六进制所需 16 个符号。  
> 2. 源码级数值书写完全脱离 `0-9 A-F`，例如 `四二` 表示 0x42，`〡〢` 表示 0xBC。  
> 3. 若字符串仅含 `〇~九` 则按十进制解析（如 `一二三` → 123），提供人类友好输入。

## 三、Hex0 中文符号加载器（信任根）

### 3.1 功能定位

Hex0 加载器是整个自举链条的**信任根**。它是一个运行于 x86-64 实模式、大小不超过 512 字节的纯机器码程序，具备以下功能：

- 通过 BIOS 中断读取键盘输入。
- 实时把用户键入的 `〇~〥` 符号转换为 4 位数值（半字节）。
- 每接收两个合法符号，组合为一个字节存入指定内存区域（如 `0x7E00`）。
- 遇回车键（Enter）停止接收，并跳至加载地址执行用户代码。
- 忽略一切非法符号，仅响应 16 个合法符号。

### 3.2 内部符号表结构

为实现 Unicode 码点 → 半字节值的映射，加载器内嵌一张查找表。出于空间效率考虑，每项仅 3 字节：

| 偏移 | 内容 | 说明 |
|:---|:---|:---|
| +0 | 码点低字节 | 因所选符号码点高字节不重复，可仅匹配低字节以简化逻辑 |
| +1 | 码点高字节 | 保留用于双字节严格匹配的版本（可选） |
| +2 | 数值 (0~15) | 对应半字节值 |

**符号表十六进制转储**（以标准十六进制表示，仅供读者参考对照, 小端）：

```
07 30 00  ; 〇 (U+3007) → 0
00 4E 01  ; 一 (U+4E00) → 1
8C 4E 02  ; 二 (U+4E8C) → 2
09 4E 03  ; 三 (U+4E09) → 3
DB 56 04  ; 四 (U+56DB) → 4
94 4E 05  ; 五 (U+4E94) → 5
6D 51 06  ; 六 (U+516D) → 6
03 4E 07  ; 七 (U+4E03) → 7
6B 51 08  ; 八 (U+516B) → 8
5D 4E 09  ; 九 (U+4E5D) → 9
41 53 0A  ; 十 (U+5341) → 10
21 30 0B  ; 〡 (U+3021) → 11
22 30 0C  ; 〢 (U+3022) → 12
23 30 0D  ; 〣 (U+3023) → 13
24 30 0E  ; 〤 (U+3024) → 14
25 30 0F  ; 〥 (U+3025) → 15
```

### 3.3 加载器核心算法伪代码

```text
算法: Hex0 中文符号加载器主环
输入: 键盘字符流
输出: 用户程序机器码写入内存 0x7E00 并跳执行

初始化:
    DS, ES, SS ← 0
    SP ← 0x7C00
    DI ← 0x7E00          // 靶内存偏移
    BX ← 0               // BH 暂存高半字节, BL=0 表示无暂存

主环:
    叫 BIOS int 0x16 读取字符 → AL
    叫 BIOS int 0x10 回显字符

    若 AL = 0x0D (回车):
        跳至 0x0000:0x7E00 执行用户代码

    若 AL = 0x08 (退格):
        若 DI > 0x7E00 则 DI ← DI - 1  // 简化处理
        继续主环

    // 在符号表中查找 AL (假设键盘可返回 Unicode 码点低字节)
    SI ← 符号表起始地址
    CX ← 16
查找环:
    AL' ← [SI]
    若 AL' = AL:
        数值 ← [SI+2]
        若 BL = 0:
            BH ← 数值
            BL ← 1       // 标记已收到高半字节
        否则:
            BH ← (BH << 4) | 数值
            ES:[DI] ← BH
            DI ← DI + 1
            BL ← 0
        跳出查找环
    SI ← SI + 3
    LOOP 查找环

    若未找到匹配，则忽略该字符
    跳回主环
```

> **说明**：真实 x86 裸机环境下 BIOS `int 0x16` 返回 ASCII/扫描码，无法直接返回 Unicode 码点。实践中需通过串口接收 UTF16BE 字节流并软件解析为码点，或使用 QEMU 自定义键盘映射模拟。此算法展示核心映射逻辑，物理实现细节超出本文范围。

## 四、中文后缀汇编句法（Forth 范式）

### 4.1 设计哲学

传统 x86 汇编采用**前缀/中缀**句法，操作码在前，操作数在后，例如：

```assembly
MOV  AX, 0x41   ; 英文: 操作码 靶, 源
```

本体系吸收 Forth 语言的**后缀表示法**与栈式计算思想，重新设计句法：

```
[操作数序列]  [操作符]
```

操作数按“先源后靶”顺序列出（与 x86 机器码编程习惯一致），最后以中文单字动词收尾。解析器维护一个操作数栈，从左向右扫描词语，遇操作数则推栈，遇操作符则弹出所需个数并生成机器码。

### 4.2 寄存器命名表

为消除单字母缩写的歧义型，寄存器全部采用功能描述型中文全称：

| x86 寄存器 | 中文全称 | 简称（可选） | 3-bit 代码 |
|:---|:---|:---|:---|
| AX | 累加器 | 累 | 000 |
| CX | 计数器 | 计 | 001 |
| DX | 数据器 | 数 | 010 |
| BX | 基址器 | 基 | 011 |
| SP | 栈指针 | 栈针 | 100 |
| BP | 基指针 | 基针 | 101 |
| SI | 源变址 | 源 | 110 |
| DI | 靶变址 | 靶 | 111 |

段寄存器：`码段(CS)`、`数段(DS)`、`附段(ES)`、`栈段(SS)`。

### 4.3 核心指令操作符集

| 语义 | 中文操作符 | 后缀句法示例 | 等效 x86 | 操作数栈弹出规则 |
|:---|:---|:---|:---|:---|
| 数据传送 | 移 | `四一 累加器 移` | `mov ax, 0x41` | (源, 靶) |
| 加法 | 加 | `〇五 累加器 加` | `add ax, 5` | (源, 靶) |
| 减法 | 减 | `一〇 累加器 减` | `sub ax, 0x10` | (源, 靶) |
| 对比 | 比 | `〇 累加器 比` | `cmp ax, 0` | (源, 靶) |
| 无条件跳 | 跳 | `环 跳` | `jmp 环` | (标号) |
| 相等跳 | 等跳 | `终 等跳` | `je 终` | (标号) |
| 中断叫 | 中断 | `一〇 中断` | `int 0x10` | (向量号) |
| 推栈 | 推 | `累加器 推` | `push ax` | (源) |

| 自增 | 增一 | `靶变址 增一` | `inc di` | (寄存器) |

| 异或 | 异或 | `累加器 累加器 异或` | `xor ax, ax` | (源, 靶) |
| 左移 | 左移 | `四 累加器 左移` | `shl ax, 4` | (位数, 靶) |
| 右移 | 右移 | `一 累加器 右移` | `shr ax, 1` | (位数, 靶) |

## 五、中文后缀宏汇编器设计

### 5.1 整体架构

汇编器采用经典**两遍扫描**架构，如图 1 所示：

```

│  (分词、去注释)│

       ▼

       │
       ▼

       ▼

└─────────────┘
```

### 5.2 词法分析器

词法分析器把输入 UTF16BE 字符流切分为 Token 序列。Token 类型定义如下表：

| Token 类型 | 正则/识别规则 | 示例 | 值表示 |
|:---|:---|:---|:---|
| 十六进制数字 | `[〇一二三四五六七八九十〡〢〣〤〥]+`，且至少包含一个 `十〡〢〣〤〥` | `四二`、`〡〢` | 整数值 |
| 十进制数字 | `[〇一二三四五六七八九]+`，不包含 `十〡〢〣〤〥` | `一二三` | 整数值 |
| 寄存器名 | 固定集合 | `累加器`、`靶变址` | 寄存器标识 |
| 操作符 | 固定集合 | `移`、`加`、`跳` | 操作符码 |
| 标号定义 | `[^ \t\n\r；]+:` | `开始:` | 标号名 |
| 标号引用 | 除上述外以汉字开头的字符串 | `开始`、`环` | 标号名 |
| 注释 | `；` 至行尾 | `； 注释` | 忽略 |

### 5.3 第一遍扫描：标号地址收集

**输入**：Token 序列  
**输出**：标号名到地址的映射表

**算法**：
1. 初始化 `地址计数器 = 0`。
2. 逐行读取 Token 序列。
3. 若行首为标号定义（如 `开始:`），记录 `标号表[标号名] = 地址计数器`。
4. 若为普通指令行，叫**模拟生成函数**计算本行会产生的字节数，并累加到地址计数器。模拟生成函数与第二遍的真实生成函数共享同一套操作符处理逻辑，但输出字节被丢弃。

**模拟生成的关键**：对于跳指令，其指令长度已知（例如 `跳` 固定 3 字节，`等跳` 固定 2 字节或 6 字节），因此可准确推进地址计数器，无需知道实际跳靶。

### 5.4 第二遍扫描：代码生成与回填

**输入**：Token 序列 + 标号表  
**输出**：纯二进制机器码字节流

**核心数据结构**：操作数栈 `stack = []`

**单行处理流程**：

```text
对于行中的每个 Token t:
    若 t 为数字常量:
        val ← 解析数字(t)
        stack.push( ('IMM', val) )
    若 t 为寄存器名:
        stack.push( ('REG', t) )
    若 t 为标号引用:
        stack.push( ('LABEL', t) )
    若 t 为操作符:
        根据操作符类型从 stack 弹出 n 个操作数
        检查操作数类型合法型
        叫对应的机器码生成函数
        把生成字节追加到输出缓冲区
        若操作符产生结果需写回寄存器，由于 x86 结果在靶操作数，无需推栈
```

**回填处理**：对于跳指令，生成代码时暂时在偏移位置填入占位符（如 `0x00 0x00`），并把（回填地址，标号名，指令类型）记录到待回填列表。全部代码生成完毕后，遍历待回填列表，根据标号表计算相对偏移并覆写占位字节。

### 5.5 关键操作符的代码生成规则

| 操作符 | 栈弹出 | 生成函数行为 |
|:---|:---|:---|
| `移` | (源, 靶) | 若靶=累加器且源为立即数，生成 `B8` + imm16 小端序；若源、靶皆为寄存器，生成 `0x89` + ModR/M 字节 (ModR/M = 0xC0 \| (reg_src<<3) \| reg_dst) |
| `加` | (源, 靶) | 若靶=累加器且源为立即数，生成 `05` + imm16 |
| `减` | (源, 靶) | 若靶=累加器且源为立即数，生成 `2D` + imm16 |
| `比` | (源, 靶) | 若靶=累加器且源为立即数，生成 `3D` + imm16 |
| `中断` | (向量号) | 生成 `CD` + imm8 |
| `跳` | (标号) | 生成 `E9` + 占位符 2 字节，记录回填信息 |
| `等跳` | (标号) | 生成 `74` + 占位符 1 字节（假设偏移在 -128~127 内），记录回填 |
| `推` | (源寄存器) | 生成 `50 + reg_code` |
| `弹` | (靶寄存器) | 生成 `58 + reg_code` |
| `腾` | 无 | 生成 `C3` |

### 5.6 数字解析模块

```python
中文十六进制映射 = { '〇':0, '一':1, '二':2, '三':3, '四':4, '五':5, '六':6, '七':7, '八':8, '九':9, '十':10, '〡':11, '〢':12, '〣':13, '〤':14, '〥':15 }

def 解析数字(字符串):
    if any(ch in 字符串 for ch in '十〡〢〣〤〥'):
        # 十六进制模式
        val = 0
        for ch in 字符串:
            val = (val << 4) | 中文十六进制映射[ch]
        return val
    else:
        # 十进制模式
        val = 0
        for ch in 字符串:
            val = val * 10 + 中文十六进制映射[ch]
        return val
```

# 全中文计算体系下哈希表实现的完整手工构造演示

## 第一部分：算法设计（纯中文描述）

### 1.1 需求分析

我们要实现一个**固定大小的哈希表**，用于存储键值对。假设：
- 表大小为 16 个槽位（索引 0~15）。
- 每个槽位存储一个**单字节键**和一个**单字节值**。空槽位用键值 `〇` 表示。
- 支持两种操作：
  1. **插入**：给定键（0~255）和值（0~255），若键已存在则更新值；否则按线型探测找到空槽插入。
  2. **查找**：给定键，返回对应的值；若键不存在，返回 `〇`。

### 1.2 数据结构

内存中布局如下（假设哈希表起始地址为 `0x2000:0x0000`，即物理地址 `0x20000`）：

```
地址偏移   内容
+0x00     槽0 的键
+0x01     槽0 的值
+0x02     槽1 的键
+0x03     槽1 的值
...
+0x1E     槽15 的键
+0x1F     槽15 的值
```

共 32 字节（16 槽 × 2 字节/槽）。

### 1.3 哈希函数

使用极简哈希函数：**键值除以 16 的余数**（即 `键 & 0x0F`，取低 4 位）。由于表大小正好为 16，此哈希函数可把任意键映射到 0~15。

### 1.4 插入算法（线型探测）

```
输入：键（AL）、值（AH）
步骤：
1. 计算哈希索引 = 键 & 0x0F → 存入 BL
2. 计算槽地址 = 表基址 + (索引 * 2)
3. 环最多 16 次：
   a. 读取该槽的键（内存字节）
   b. 若该键 == 0（空槽）或 == 输入键：
       把值写入该槽的值字段
       把输入键写入该槽的键字段
       终
   c. 否则（发生冲突）：
       索引 = (索引 + 1) & 0x0F
       更新槽地址
       继续环
4. 若环终仍未找到空槽/相同键，则表满，操作失败（此处忽略失败处理）。
```

### 1.5 查找算法

```
输入：键（AL）
输出：值（AL）
步骤：
1. 计算哈希索引 = 键 & 0x0F → 存入 BL
2. 计算槽地址 = 表基址 + (索引 * 2)
3. 环最多 16 次：
   a. 读取该槽的键（内存字节） → CL
   b. 若 CL == 0（空槽）：
       返回值 = 0
       终
   c. 若 CL == 输入键：
       读取该槽的值 → AL
       终
   d. 否则：
       索引 = (索引 + 1) & 0x0F
       更新槽地址
       继续环
4. 若环终仍未找到，返回值 = 0（表满且未命中，实际不会执行到）。
```

我们使用此前定义的中文后缀句法，并假设以下伪指令已由汇编器支持：
- `内存读 字节`：从内存地址读取一个字节到 AL（对应 x86 `mov al, [bx]` 等，此处简化用“读”操作符）。
- `内存写 字节`：把 AL 写入内存地址（对应 x86 `mov [bx], al`）。

为了简化，我们采用一个模拟的栈式虚拟机指令集，但手工翻译时会映射到真实的 x86 指令。下面给出完整汇编源码，注释详细。
